# Amazon EC2 - Elastic Compute Cloud

> **Amazon EC2 (Elastic Compute Cloud)** 是 AWS 的核心计算服务，提供安全、可调整的虚拟服务器容量。它是 AWS 云中最基础和最常用的计算服务。
>
> 相关文档：[[块储存 - EBS]] | [[文件储存 - AWS EFS]] | [[Security Group]] | [[Auto Scaling]] | [[IAM]]

---

## 核心概念

### EC2 定位

| 服务 | 类型 | 场景 | 特点 |
|------|------|------|------|
| **EC2** | 虚拟机 | 传统应用托管 | 完全控制服务器 |
| [[Lambda]] | Serverless 计算 | 事件驱动函数执行 | 无需管理服务器 |
| [[ECS]] | 容器 | 微服务部署 | 容器编排 |
| [[AWS Fargate]] | Serverless 容器 | 无服务器容器 | 无需管理集群 |

### EC2 核心组件

| 组件 | 描述 |
|------|------|
| **AMI (Amazon Machine Image)** | 实例的模板，包含 OS、应用服务器、软件 |
| **Instance (实例)** | EC2 虚拟机实例 |
| **Instance Type** | 定义 CPU、内存、存储、网络资源配置 |
| **Security Group** | 虚拟防火墙，控制入站/出站流量 |
| **Key Pair** | 用于安全登录的公钥/私钥对 |
| **IAM Role** | 分配给实例的权限角色 |

## 实例类型

### 实例类别对比（考试重点）

| 类别 | 代表型号 | 特点 | 适用场景 |
|------|---------|------|---------|
| **通用型 (General Purpose)** | T3, M5, M6g | 平衡计算、内存、网络资源 | Web 服务器、中小型数据库、开发测试 |
| **计算优化型 (Compute Optimized)** | C5, C6g | 高 CPU 比率 | 高性能 Web 服务器、机器学习、批处理 |
| **内存优化型 (Memory Optimized)** | R5, X1e, Z1dn | 大内存容量 | 内存数据库 (Redis)、实时大数据分析 |
| **加速计算型 (Accelerated Computing)** | P3, G4, P4 | 硬件加速器 (GPU/FPGA) | 图形渲染、机器学习训练、科学计算 |
| **存储优化型 (Storage Optimized)** | I3, D2, H1 | 高顺序/随机 I/O | NoSQL 数据库、数据仓库、OLAP |

## 存储选项

### 存储类型对比

| 存储类型 | 持久性 | 速度 | 适用场景 |
|---------|--------|------|---------|
| **EBS (Elastic Block Store)** | 持久化 | 高 | 操作系统盘、数据库、企业级应用 |
| **实例存储 (Instance Store)** | 临时性 | 极高 | 缓存、临时数据处理、分析 |

| 卷类型 | 类型 | 适用场景 | IOPS |
|--------|------|---------|------|
| **io2 / io1** | SSD (PIOPS) | 关键业务数据库 | 最高 256,000 |

| **gp2** | SSD (通用) | 中小规模工作负载 | 1-10,000 |
| **st1** | HDD | 大数据、数据仓库 | 最高 500 |
| **sc1** | HDD | 冷数据归档 | 最高 250 |

## 安全与访问控制

### 安全组 (Security Group)

- 充当**虚拟防火墙**，控制实例的入站和出站流量
- **有状态**：允许的请求响应会自动放行
- 支持允许规则，不支持拒绝规则
- 关联到实例级别
- 相关文档：[[Security Group]]、[[NACL]]

### 密钥对 (Key Pairs)

- 使用**公钥/私钥对**进行身份验证
- 用于加密登录凭据，确保远程登录安全
- 支持 PEM（Linux/Mac）和 PPK（Windows）格式

### IAM 角色

- 将 IAM 角色分配给实例，使应用程序能够**安全地访问其他 AWS 资源**
- 无需在代码中硬编码访问密钥
- 自动管理临时凭证
- 相关文档：[[IAM]]

#### 如何扩展 EC2 实例权限（考试重点）

> **在 AWS 中，为正在运行的实例增加权限无需重新创建实例。**

| 方法 | 操作 | 效果 |
|------|------|------|
| **原地升级旧角色** | 在现有 IAM 角色中追加新的 IAM 策略（如 API Gateway 访问权限） | 实例通过**实例元数据 (Instance Metadata)** 获取的临时凭证会自动更新权限 |
| **替换新角色** | 创建全新角色，挂载 S3 和 API Gateway 的策略，然后将实例关联的角色从旧角色替换为新角色 | 一个 EC2 实例在同一时间只能关联**一个 IAM 角色**（即实例配置文件 Instance Profile） |

#### 重要限制

- **一个 EC2 实例 = 一个 IAM 角色**：不能通过"叠加"多个角色来增加权限
- **通过策略扩展权限**：在一个角色下挂载多个策略来实现权限扩展
- **非破坏性修改**：AWS 支持在运行中的实例上附加、更换或取下 IAM 角色
- **运维效率**：修改 IAM 配置是零停机的，重新创建实例涉及停机时间和数据迁移

#### 场景题解题思路

```
场景分析 → 扩展 EC2 权限
├── "需要增加权限" → 修改 IAM 角色策略
├── "需要多个服务权限" → 创建一个包含多个策略的角色
├── "重新创建实例" → ❌ 错误（非破坏性、高成本）
└── "叠加多个角色" → ❌ 错误（一个实例只能有一个角色）
```

#### 常见考试场景分析

**场景：EC2 实例需要访问 S3 和 API Gateway，当前只关联了 S3 访问权限的角色，如何扩展？**

| 选项 | 是否正确 | 原因 |
|------|---------|------|
| **修改 IAM 角色策略** | ✅ 正确 | 零停机，追加 API Gateway 访问策略即可 |
| **替换为新角色** | ✅ 正确 | 创建包含 S3 + API Gateway 策略的新角色 |
| 重新创建实例 | ❌ 错误 | 非破坏性修改，重新创建代价极大 |
| 叠加多个角色 | ❌ 错误 | 一个实例只能关联一个 IAM 角色 |

**关键总结：**
- **改角色里的策略** → 追加新策略
- **给实例换个新角色** → 包含所有需要的策略
- **重新创建实例** → ❌ 绝对错误
- **叠加多个角色** → ❌ 不支持

### 网络 ACL (NACL)

- 无状态防火墙，关联到子网
- 支持允许和拒绝规则
- 作为安全组的额外保护层

## 定价模式

### 定价选项对比（考试重点）

| 定价模式 | 灵活性 | 成本 | 适用场景 | 最小单位 |
|---------|--------|------|---------|---------|
| **On-Demand** | 最高 | 标准价格 | 短期、不可预测的工作负载 | 秒 |
| **Spot** | 最低 | 最高 90% 折扣 | 容错性高的任务、批处理 | 秒 |
| **Reserved (RS)** | 低 | 最高 72% 折扣 | 长期稳定运行的应用 | 1 年/3 年 |
| **Savings Plans** | 中 | 最高 66% 折扣 | 承诺特定金额的使用量 | 1 年/3 年 |

### On-Demand 详细说明

- **灵活性**：无需承诺，随时启动/终止
- **区域限制**：每个区域实例限制通常为 20 台（可申请提升）
- **计费单位**：秒（部分实例类型按小时）
- **适用场景**：突发流量、短期测试、不确定 workload

### Spot Instance 详细说明

- **成本**：利用未使用的 EC2 容量，最高 90% 折扣
- **中断风险**：AWS 可在 2 分钟前中断实例
- **适用场景**：
  - 批处理作业
  - 数据分析
  - 渲染农场
  - 容错性高的无状态应用
- **Spot Fleet**：管理一组 Spot 和 On-Demand 实例

### Reserved Instance 详细说明

- **承诺期**：1 年（保守）或 3 年（更大折扣）
- **类型**：
  - **标准 RI**：最高折扣，交换受限
  - **可转换 RI**：可交换实例类型
  - **部分预付 RI**：前期支付部分费用
- **适用范围**：特定实例类型和区域

## 置放群组 (Placement Groups)

> **置放群组** 决定了 EC2 实例在底层硬件上的物理分布方式，直接影响高可用性和网络性能。

### 三种置放群组类型

| 类型 | 特点 | 适用场景 | 故障影响 |
|------|------|---------|---------|
| **Cluster (集群)** | 实例放置在物理上极其接近的硬件中（同一机架） | 极致网络性能、低延迟 | 高 - 底层硬件故障影响整个群组 |
| **Spread (分散)** | 实例分布在不同的底层硬件机架上 | 高可用性、减少故障影响 | 低 - 每个实例独立硬件 |
| **Partition (分区)** | 实例分布在不同的分区（每组约 7 个机架） | HDFS、HBase、Cassandra | 中 - 单个分区故障不影响其他分区 |

### Cluster vs Spread 对比（考试高频考点）

| 特性 | Cluster | Spread |
|------|---------|--------|
| **网络延迟** | 最低（同一机架） | 较高（不同机架） |
| **高可用性** | 低（故障关联） | 高（物理隔离） |
| **实例数量限制** | 无限制 | 每 AZ 最多 7 个运行中实例 |
| **成本** | 无额外费用 | 无额外费用 |
| **需求关键词** | "极致网络性能"、"低延迟" | "减少硬件故障影响"、"高可用" |

### 场景题解题思路

```
场景分析 → 选择置放群组类型
├── "需要极致网络性能/低延迟" → Cluster
├── "需要减少硬件故障影响" → Spread
├── "需要大规模分布式存储" → Partition
└── "14 个实例，高可用" → Spread × 2 AZ (7×2=14)
```

### 常见考试场景分析

**场景：需要部署 14 个实例，减少硬件节点故障的影响，最具成本效益的方案？**

| 选项 | 是否正确 | 原因 |
|------|---------|------|
| **Spread 置放群组** | ✅ 正确 | 物理隔离，每 AZ 最多 7 个实例，需 2 个 AZ（7×2=14） |
| Cluster 置放群组 | ❌ 错误 | 实例物理靠拢，硬件故障影响整个群组 |
| Dedicated Hosts | ❌ 错误 | 费用昂贵，仅用于合规或软件许可场景 |
| 默认置放 | ❌ 错误 | 无法保证物理隔离 |

**关键总结：**
- **降低硬件故障风险** → 选 **Spread**
- **极致网络性能** → 选 **Cluster**
- **置放群组本身免费**，只需支付常规实例费用

## AMI (Amazon Machine Image)

### AMI 来源

| 来源 | 描述 | 示例 |
|------|------|------|
| **Quick Start** | AWS 提供的官方镜像 | Amazon Linux 2, Ubuntu |
| **市场 (Marketplace)** | 第三方提供的镜像 | RHEL, SUSE, 商业软件 |
| **社区** | 其他用户共享的镜像 | 社区 AMI |
| **自定义** | 从现有实例或容器镜像创建 | 私有应用镜像 |

### AMI 生命周期

```
启动实例 → 配置软件 → 创建 AMI → 从 AMI 启动新实例
```

- **创建 AMI**：可包含 EBS 快照
- **共享 AMI**：可共享给其他 AWS 账户或公开
- **复制 AMI**：可复制到另一个区域

## 实例生命周期

### 实例状态

| 状态 | 描述 |
|------|------|
| **pending** | 实例正在启动 |
| **running** | 实例正在运行 |
| **stopping** | 实例正在停止 |
| **stopped** | 实例已停止（EBS 保留，可按需启动） |
| **shutting-down** | 实例正在终止 |
| **terminated** | 实例已终止 |

### 启动和停止 vs 终止

| 操作 | EBS 卷 | 公网 IP | 计费 | 适用场景 |
|------|--------|--------|------|---------|
| **停止 (Stop)** | 保留 | 释放 | 不按计算计费 | 短期暂停 |
| **终止 (Terminate)** | 按 DeleteOnTermination 设置 | 释放 | 停止计费 | 永久删除 |

## 高可用性设计

### 可用区部署

- EC2 实例分布在**多个可用区 (AZ)** 实现故障容错
- 结合 **ELB (Elastic Load Balancer)** 实现负载均衡
- 结合 **Auto Scaling** 实现自动故障恢复

### 跨区域部署

- 在**多个区域**部署实例实现灾难恢复
- 使用 **Route 53** 实现 DNS 级别故障转移
- 使用 **S3 Cross-Region Replication** 同步数据

## 考试重点总结

### SAA-C02 高频考点

1. **实例类型选择**：根据工作负载选择正确的实例类别
2. **定价模式**：On-Demand（灵活）、Spot（便宜）、Reserved（长期）、Savings Plans（灵活承诺）
3. **置放群组**：Cluster（性能）、Spread（高可用）、Partition（分布式存储）
4. **EBS vs 实例存储**：持久化 vs 临时性
5. **安全组 vs NACL**：有状态 vs 无状态
6. **AMI**：实例模板，可用于快速部署
7. **停止 vs 终止**：EBS 保留 vs 按策略删除
8. **Auto Scaling**：水平扩展能力
9. **IAM 角色**：安全访问其他 AWS 资源
10. **实例命名规则**：世代 + 类别 + 大小

### 场景题解题思路

```
场景分析 → 选择 EC2 配置
├── "短期、不可预测" → On-Demand
├── "容错性高、批处理" → Spot Instance
├── "长期稳定运行" → Reserved Instance
├── "极致网络性能" → Cluster 置放群组
├── "高可用性" → Spread 置放群组 + 多 AZ
├── "内存数据库" → R5/X1e 内存优化型
└── "图形渲染" → P3/G4 加速计算型
```

## 最佳实践

1. **选择合适的实例类型**：根据工作负载选择最优类别
2. **使用 Spot Instance 降低成本**：容错性高的任务使用 Spot
3. **多 AZ 部署**：提高应用可用性
4. **启用 EBS 加密**：保护静态数据
5. **使用 IAM 角色**：避免硬编码访问密钥
6. **配置 Auto Scaling**：自动应对流量变化
7. **定期更新 AMI**：保持系统和应用最新
8. **监控实例性能**：使用 CloudWatch 监控指标
