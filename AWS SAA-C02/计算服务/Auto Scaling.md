# Auto Scaling - 弹性伸缩

> **Auto Scaling** 是一种自动管理 EC2 实例容量的服务，根据需求自动调整实例数量，平衡可用性、性能和成本。
>
> 相关文档：[[CloudWatch]] | [[AWS Load Balance]] | [[EC2]]

---

## 核心概念

### Auto Scaling 定位

| 服务 | 类型 | 场景 | 特点 |
|------|------|------|------|
| **Auto Scaling** | 容量管理 | 自动调整实例数量 | 根据需求伸缩 |
| **Elastic Load Balancing** | 负载均衡 | 分发流量到多个实例 | 高可用、故障转移 |
| **EC2** | 虚拟机 | 计算资源 | 实例基础 |
| **CloudWatch** | 监控 | 指标收集和告警 | 伸缩触发条件 |

### 核心组件

| 组件 | 描述 |
|------|------|
| **启动模板 (Launch Template)** | 定义实例配置（AMI、类型、密钥、安全组、User Data 等） |
| **启动配置 (Launch Configuration)** | 旧版，功能有限，推荐使用启动模板 |
| **Auto Scaling Group (ASG)** | 定义实例数量范围和伸缩策略 |

### ASG 容量设置

| 参数 | 描述 |
|------|------|
| **最小容量 (Min Size)** | ASG 中始终保持的最少实例数 |
| **最大容量 (Max Size)** | ASG 中允许的最大实例数 |
| **期望容量 (Desired Capacity)** | ASG 希望维持的实例数 |

## 扩展策略

### 动态扩展（基于 CloudWatch 告警）

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| **简单扩展 (Simple Scaling)** | 触发警报后增加/减少实例数，进入冷却期 | 流量变化平缓 |
| **步进扩展 (Step Scaling)** | 根据指标超出阈值的程度进行扩展 | 应对剧烈流量波动 |
| **目标追踪 (Target Tracking)** | 设定目标值（如 CPU 50%），系统自动调整容量 | 大多数场景（推荐） |

### 扩展策略对比

| 特性 | 简单扩展 | 步进扩展 | 目标追踪 |
|------|---------|---------|---------|
| 配置复杂度 | 低 | 中 | 低 |
| 响应速度 | 慢 | 快 | 快 |
| 成本效率 | 中 | 高 | 高 |
| 适用场景 | 简单负载 | 剧烈波动 | 稳定目标 |

### 计划扩展

- 基于可预测的负载模式
- 在需求发生前预先扩展
- 适用于：工作日高峰、促销活动

### 手动扩展

- 通过手动修改 ASG 的期望容量
- 适用于：临时需求、维护操作

## 高可用和容错

### 与负载均衡集成

- Auto Scaling 与 **Elastic Load Balancer (ELB)** 紧密集成
- ASG 利用 ALB 健康检查判断实例是否正常运行
- 实例不健康时自动替换

### 健康检查宽限期 (Health Check Grace Period)

> 不是用来检测故障，而是给新启动实例预留"启动和配置"的缓冲时间。

| 决定因素 | 说明 |
|---------|------|
| **引导脚本运行时间** | User Data 脚本安装软件、更新补丁或配置环境的时间 |
| **AMI 预热时间** | 代码预先安装在 AMI 中（黄金镜像）启动快，代码需下载时启动慢 |

```
实例启动 → 运行 User Data 脚本 → 健康检查通过 → 开始接收流量
          ↑
    宽限期保护（防止误杀正在启动的实例）
```

### 健康检查类型

| 类型 | 说明 |
|------|------|
| **EC2 健康检查** | 检查实例状态（系统/实例状态） |
| **ELB 健康检查** | 检查应用层健康（HTTP/TCP） |
| **自定义健康检查** | 使用 CloudWatch 自定义指标 |

## 故障排查操作

### Standby 模式（保护特定实例）

> **将实例置于 Standby 状态是故障排查的最佳实践**

| 特性 | 说明 |
|------|------|
| **移出目标组** | 实例从负载均衡器目标组中移除，不再接收流量 |
| **保持运行** | 实例不被终止，可用于调试 |
| **ASG 逻辑调整** | ASG 临时减小期望容量，不会触发替换 |

#### 为什么选择 Standby？

| 原因 | 说明 |
|------|------|
| **保持运行供调试** | 实例不被终止，可 SSH 登录排查问题 |
| **ASG 逻辑调整** | ASG 临时减小期望容量，不会触发新实例替换 |

#### Standby vs 其他操作对比

| 操作 | ASG 行为 | 是否推荐 |
|------|---------|---------|
| **Standby** | 实例保护，不触发替换 | ✅ 推荐 |
| **从目标组移除** | ASG 仍认为实例属于组内，可能触发替换 | ❌ 不推荐 |
| **终止实例** | ASG 立即启动新实例替换 | ❌ 无法调试 |
| **停止实例** | ASG 检测到不健康，启动新实例 | ❌ 不推荐 |

### 暂停扩展流程（Suspend Scaling Processes）

> **暂停扩展流程可冻结 ASG 的自动化逻辑**

| 暂停类型 | 说明 |
|---------|------|
| **Launch** | 阻止启动新实例 |
| **Terminate** | 阻止终止现有实例 |
| **AddToLoadBalancer** | 阻止实例添加到负载均衡器 |
| **AlarmNotification** | 阻止告警通知触发扩展 |
| **ScheduledActions** | 阻止计划扩展动作 |
| **HealthCheck** | 阻止健康检查 |
| **ReplaceUnhealthy** | 阻止替换不健康实例 |
| **ScheduledInstances** | 阻止计划实例 |

#### 为什么需要暂停扩展？

| 原因 | 说明 |
|------|------|
| **冻结自动化** | 彻底阻止 ASG 的自动伸缩逻辑 |
| **防止干扰** | 避免动态伸缩策略因 CPU 波动等产生干扰 |
| **环境静止** | 确保排查期间环境不变，避免不必要的费用 |

### 故障排查最佳组合

```
Standby + Suspend = 最佳故障排查方案

Standby：保护该特定实例不被删除且不接客
Suspend：关掉整个组的自动化大脑
```

#### 场景题解题思路

**场景：需要排查 ASG 中某个实例的问题，不希望该实例被替换，也不希望 ASG 自动伸缩？**

| 选项 | 是否正确 | 原因 |
|------|---------|------|
| **Standby + Suspend** | ✅ 正确 | Standby 保护实例，Suspend 阻止自动化 |
| 仅从目标组移除 | ❌ 错误 | ASG 仍认为实例属于组内，可能触发替换 |
| 仅终止实例 | ❌ 错误 | ASG 立即启动新实例，无法调试 |
| 仅停止实例 | ❌ 错误 | ASG 检测到不健康，启动新实例 |

**关键总结：**
- **保护特定实例** → Standby
- **阻止整个组自动化** → Suspend
- **排查期间保持环境静止** → Standby + Suspend

## 成本优化

### 扩展优化

| 策略 | 说明 |
|------|------|
| **Scaling In** | 需求下降时减少容量，优化成本 |
| **使用 Spot 实例** | 容错性高的任务使用 Spot |
| **设置合理的最大容量** | 防止意外扩展导致高费用 |
| **使用目标追踪** | 自动维持目标资源利用率 |

### 冷却期 (Cooldown)

- 扩展操作后等待一段时间再进行下一次扩展
- 让指标稳定，避免过度扩展
- 默认 300 秒

## 区域特性

### 管理边界

- ASG 绑定在**单个 AWS 区域**内
- 不能创建跨区域的 ASG

### 跨可用区部署

- ASG 可横跨同一区域内的**多个可用区 (AZ)**
- 一个 AZ 故障时，ASG 自动在其他健康 AZ 启动新实例
- 推荐配置：至少 2-3 个 AZ

```
区域：us-east-1
├── AZ: us-east-1a (3 实例)
├── AZ: us-east-1b (3 实例)
└── AZ: us-east-1c (3 实例)
总期望容量：9 实例
```

## 考试重点总结

### SAA-C02 高频考点

1. **扩展策略选择**：简单扩展（简单）、步进扩展（剧烈波动）、目标追踪（大多数场景）
2. **冷却期**：防止过度扩展，让指标稳定
3. **健康检查宽限期**：给新实例启动时间，防止误杀
4. **Standby 模式**：保护特定实例，用于故障排查
5. **暂停扩展流程**：冻结 ASG 自动化，用于故障排查
6. **跨 AZ 部署**：高可用，一个 AZ 故障时自动切换
7. **与 ELB 集成**：利用 ELB 健康检查判断实例健康
8. **ASG 容量设置**：Min、Max、Desired 的关系
9. **计划扩展**：基于可预测负载模式
10. **Auto Scaling 免费**：服务本身免费

### 场景题解题思路

```
场景分析 → 选择 ASG 配置
├── "流量平缓变化" → 简单扩展
├── "剧烈流量波动" → 步进扩展
├── "维持目标 CPU" → 目标追踪
├── "可预测负载" → 计划扩展
├── "排查特定实例" → Standby
├── "阻止自动化" → Suspend
├── "高可用" → 多 AZ 部署
└── "降低成本" → Spot 实例 + Scaling In
```

## 最佳实践

1. **使用目标追踪**：大多数场景下最简单高效
2. **多 AZ 部署**：至少 2-3 个 AZ 实现高可用
3. **设置健康检查宽限期**：给新实例足够启动时间
4. **使用 ELB 健康检查**：比 EC2 健康检查更准确
5. **使用启动模板**：替代旧的启动配置
6. **设置合理的最大容量**：防止意外扩展导致高费用
7. **使用 Spot 实例**：容错性高的任务使用 Spot
8. **定期审查扩展策略**：根据实际负载调整
