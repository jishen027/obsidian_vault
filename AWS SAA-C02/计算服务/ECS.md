# Amazon ECS - Elastic Container Service

> **Amazon ECS (Elastic Container Service)** 是一种高度可扩展、高性能的容器管理服务，允许您在 AWS 上轻松运行、停止和管理 Docker 容器。
>
> 相关文档：[[EC2]] | [[AWS Fargate]] | [[Auto Scaling]] | [[AWS Load Balance]]

---

## 核心概念

### ECS 定位

| 服务              | 类型            | 场景       | 特点        |
| --------------- | ------------- | -------- | --------- |
| **ECS**         | 容器编排          | 微服务部署    | 容器编排、任务管理 |
| [[AWS Fargate]] | Serverless 容器 | 无服务器容器   | 无需管理集群    |
| [[EC2]]     | 虚拟机           | 传统应用托管   | 完全控制服务器   |
| [[Lambda]]      | Serverless 计算 | 事件驱动函数执行 | 无需管理服务器   |

### ECS 核心组件

| 组件 | 描述 |
|------|------|
| **任务定义 (Task Definition)** | JSON 格式的蓝图，描述容器的配置（镜像、内存、CPU、端口等） |
| **任务 (Task)** | 任务定义的运行实例 |
| **服务 (Service)** | 维护预定数量的任务副本，与负载均衡器集成 |
| **集群 (Cluster)** | 逻辑分组，用于管理任务和服务的计算资源 |

## 两种运行模式（考试重点）

### Fargate vs EC2 模式对比

| 特性 | Fargate（无服务器） | EC2 模式 |
|------|-------------------|---------|
| **基础设施管理** | 无需管理 | 自行管理 EC2 实例 |
| **成本模型** | 按容器资源使用计费 | 按 EC2 实例计费 |
| **启动速度** | 快（几秒钟） | 慢（需要启动 EC2） |
| **成本控制** | 较高 | 可优化（Spot 实例、预留实例） |
| **适用场景** | 微服务、突发负载 | 长期稳定负载、GPU 需求 |
| **推荐程度** | **现代应用推荐** | 特定需求时使用 |

### 选择指南

```
选择运行模式
├── "追求极简、不想管服务器" → Fargate
├── "需要 GPU" → EC2 模式
├── "优化成本、长期使用" → EC2 模式（Spot 实例）
├── "特定底层硬件需求" → EC2 模式
└── "微服务、突发负载" → Fargate
```

## 架构详解

### 组件关系

```
集群 (Cluster)
├── 服务 (Service)
│   ├── 任务 (Task) → 容器 (Container)
│   └── 任务 (Task) → 容器 (Container)
└── 任务 (Task) → 容器 (Container)
```

### 任务定义 (Task Definition)

| 配置项 | 说明 |
|--------|------|
| **容器镜像** | Docker 镜像地址（通常来自 ECR） |
| **内存限制** | 容器可用的内存（MB/GB） |
| **CPU 限制** | 容器可用的 vCPU 数量 |
| **端口映射** | 容器端口到主机端口的映射 |
| **IAM 角色** | 任务级别的权限 |
| **日志配置** | CloudWatch Logs 集成 |

### 服务 (Service)

| 功能 | 说明 |
|------|------|
| **任务副本管理** | 维护预定数量的任务运行中 |
| **负载均衡集成** | 与 ALB/NLB 集成处理流量 |
| **健康检查** | 自动替换不健康的任务 |
| **滚动更新** | 支持任务定义的滚动更新 |

## ECS 优势

### 核心优势

| 优势 | 说明 |
|------|------|
| **快速启动** | 容器几秒钟内启动，比 EC2 快数分钟 |
| **资源利用率高** | "装箱"技术，单实例部署多个微服务 |
| **深度集成** | 与 ECR、IAM、CloudWatch 集成 |
| **可扩展性** | 自动扩展任务数量 |
| **安全性** | IAM 角色为每个任务分配精细权限 |

### 与 ECR 集成

- **ECR (Elastic Container Registry)**：AWS 专用的容器镜像注册表
- 安全存储和拉取容器镜像
- 与 ECS 无缝集成
- 支持镜像扫描和漏洞检测

### 与 IAM 集成

- 任务级别的 IAM 角色
- 精细权限控制
- 无需在容器内硬编码访问密钥

## 典型应用场景

| 场景 | 推荐配置 | 说明 |
|------|---------|------|
| **微服务架构** | ECS + Fargate | 独立运行、互相解耦的容器化组件 |
| **批处理作业** | ECS + EC2（Spot 实例） | 短时间内大量计算资源 |
| **Web 应用** | ECS + ALB | 与负载均衡器集成处理流量 |
| **混合部署** | ECS Anywhere | 云端容器管理扩展到本地数据中心 |

## 与相关服务对比

### ECS vs Fargate vs Lambda

| 特性 | ECS | Fargate | Lambda |
|------|-----|---------|--------|
| **服务器管理** | 可选（EC2 模式） | 无需管理 | 无需管理 |
| **执行时间** | 无限制 | 最长 14 天 | 最长 15 分钟 |
| **冷启动** | 快（秒级） | 快（秒级） | 可能有冷启动延迟 |
| **成本** | 按 EC2 + 资源 | 按资源使用 | 按请求 + 执行时间 |
| **适用场景** | 长期运行的容器 | 短期任务、微服务 | 事件驱动函数 |

### ECS vs EKS

| 特性 | ECS | EKS |
|------|-----|-----|
| **编排器** | AWS 管理 | Kubernetes |
| **学习曲线** | 低 | 高（需要 K8s 知识） |
| **兼容性** | 仅 Docker | 标准 K8s 兼容 |
| **适用场景** | 简单容器管理 | 复杂 K8s 工作负载 |

## 限制和配额

| 限制项 | 默认值 |
|--------|--------|
| **每个区域的集群数量** | 无硬限制 |
| **每个服务的任务数量** | 默认 1000（可申请提升） |
| **任务定义数量** | 默认 100（可申请提升） |
| **容器镜像大小** | ECR 支持最大 20 GB |

## 定价模型

### 计费项

| 计费项 | 说明 |
|--------|------|
| **ECS 服务** | **免费**，只需支付底层资源费用 |
| **Fargate** | 按容器 vCPU 和内存使用量计费 |
| **EC2 实例** | 按运行时间计费 |
| **ECR 镜像存储** | 按存储量计费 |
| **ALB/NLB** | 按运行时间和负载均衡容量单位计费 |

> **重要**：ECS 服务本身是免费的！

## 考试重点总结

### SAA-C02 高频考点

1. **ECS 定位**：容器编排服务，管理 Docker 容器
2. **Fargate vs EC2**：无服务器 vs 自管理，现代应用推荐 Fargate
3. **核心组件**：任务定义、任务、服务、集群
4. **与 ECR 集成**：安全存储和拉取容器镜像
5. **与 IAM 集成**：任务级别权限控制
6. **快速启动**：容器秒级启动，比 EC2 快
7. **资源利用**：装箱技术，提高硬件利用率
8. **负载均衡集成**：与 ALB/NLB 集成处理流量
9. **ECS 免费**：服务本身免费
10. **ECS vs Fargate**：ECS 是编排器，Fargate 是计算驱动方式

### 场景题解题思路

```
场景分析 → 选择容器方案
├── "微服务、不想管服务器" → ECS + Fargate
├── "GPU 需求" → ECS + EC2 模式
├── "优化成本、长期负载" → ECS + EC2（Spot 实例）
├── "事件驱动函数" → Lambda
├── "K8s 兼容性" → EKS
└── "容器镜像存储" → ECR
```

### 常见考试场景分析

**场景：公司需要部署微服务架构，追求极简管理，不想管服务器，最佳方案？**

| 选项 | 是否正确 | 原因 |
|------|---------|------|
| **ECS + Fargate** | ✅ 正确 | 无服务器容器，无需管理基础设施 |
| ECS + EC2 模式 | ❌ 错误 | 需要管理 EC2 实例，增加运维负担 |
| EC2 | ❌ 错误 | 需要自行管理容器编排 |
| Lambda | ❌ 错误 | 适合事件驱动函数，不适合长期运行的微服务 |

**关键总结：**
- **"微服务、不想管服务器"** → ECS + Fargate
- **"GPU 需求"** → ECS + EC2 模式
- **"事件驱动函数"** → Lambda
- **"容器镜像存储"** → ECR
