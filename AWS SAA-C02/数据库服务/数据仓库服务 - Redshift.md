# Amazon Redshift - 数据仓库服务

> **Amazon Redshift** 是一种完全托管的、基于 PB 级数据的数据仓库服务，专为联机分析处理（OLAP）场景优化。它针对海量数据集运行复杂、计算密集型查询而设计。
>
> 相关文档：[[关系型数据库 - RDS]] | [[S3 - Object Storage]] | [[DynamoDB - NoSQL]]

---

## 核心概念

### Redshift 定位

| 服务 | 类型 | 场景 | 特点 |
|------|------|------|------|
| [[关系型数据库 - RDS]] | OLTP | 日常事务处理 | 频繁读写，低延迟，行式存储 |
| **Redshift** | OLAP | 商业智能、数据分析 | 复杂查询，批量分析，列式存储 |
| [[DynamoDB - NoSQL]] | NoSQL | 键值/文档存储 | 单毫秒延迟，高吞吐，弹性扩展 |

### OLTP vs OLAP 对比（考试重点）

| 特性 | OLTP (RDS) | OLAP (Redshift) |
|------|-----------|----------------|
| 用途 | 日常事务处理 | 商业智能、数据分析 |
| 查询类型 | 简单、频繁 | 复杂、批量 |
| 数据量 | 中小规模 | PB 级 |
| 存储方式 | 行式存储 | 列式存储 |
| 事务处理 | ACID | 大部分 ACID |
| 典型操作 | INSERT, UPDATE, DELETE | SELECT, 聚合分析 |

## 架构设计

### 集群架构

```
                    ┌─────────────────────────┐
                    │     Leader Node          │
                    │  (查询解析、优化、协调)   │
                    └───────────┬─────────────┘
                                │
                ┌───────────────┼───────────────┐
                │               │               │
        ┌───────┴──────┐ ┌─────┴──────┐ ┌─────┴──────┐
        │  Compute Node│ │Compute Node│ │Compute Node│
        │  (数据处理)   │ │ (数据处理)  │ │ (数据处理)  │
        └──────────────┘ └────────────┘ └────────────┘
```

| 组件 | 描述 |
|------|------|
| **Leader Node** | 负责查询解析、优化、计划生成和结果汇总 |
| **Compute Node** | 执行实际数据处理和查询操作 |
| **Slice (切片)** | 每个 Compute Node 上的数据分区，实现并行处理 |

### 节点类型

| 节点类型 | 存储介质 | 特点 | 适用场景 |
|---------|---------|------|---------|
| **DS2 (Dense Storage)** | 磁性磁盘 (HDD) | 大容量、低成本 | 海量数据存储（最大 64 PB） |
| **DC2 (Dense Compute)** | SSD | 高性能、快速计算 | 计算密集型查询 |

### 节点规模

- **最小配置**：1 个节点（开发测试）
- **最大配置**：8,000+ 个节点（超大规模部署）
- **节点间复制**：数据在集群内节点间自动复制，提供容错能力

## 存储优化

### 列式存储

- 将同一列的值存储在一起，而非整行存储
- **优势**：
  - 减少 I/O 开销：只读取需要的列
  - 提高压缩率：同类型数据压缩效果更好
  - 加速聚合查询：SUM、AVG、COUNT 等操作更快

### 数据分布键（Distribution Keys）

> **分布策略直接影响查询性能，是 SAA-C02 考试高频考点**

| 分布策略 | 说明 | 适用场景 | 优点 | 缺点 |
|---------|------|---------|------|------|
| **KEY** | 根据指定列的值将相关数据存储在同一节点 | 大表 JOIN 小表 | 减少数据移动 | 数据倾斜风险 |
| **ALL** | 将表完整副本分发到所有节点 | 小表（维度表） | 无数据移动 | 占用更多存储 |
| **EVEN** | 均匀分布到所有节点 | 默认策略 | 数据均匀分布 | JOIN 操作可能跨节点 |

### 排序键（Sort Keys）

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| **复合排序键** | 指定多列作为排序键 | 常用过滤列 |
| **自动排序** | Redshift 自动选择排序列 | 不想手动配置 |
| **未排序表** | 不指定排序键 | 临时表、频繁写入 |

### 压缩编码

- Redshift 自动压缩数据以节省存储空间
- 压缩类型：
  - **RAW**：不压缩
  - **MOST COMPRESSED**：最大压缩
  - **BYTEDICT / RUNLENGTH / EVEN / ODD / TEXT32 / TEXT256**：根据数据类型自动选择
- **考试提示**：启用压缩可减少存储成本和 I/O，提升查询性能

## 性能优化

### 并行处理

- Redshift 将查询分解为多个子任务，在多个节点上并行执行
- **MPP (Massively Parallel Processing)** 架构
- 数据在 Slice 间分布，实现数据局部性处理

### 查询优化

| 优化技术 | 说明 |
|---------|------|
| **查询缓存** | 常用查询结果缓存 |
| **查询队列** | 自动管理并发查询 |
| **查询监控规则** | 实时监控和告警 |
| **工作负载管理 (WLM)** | 为不同查询分配不同优先级 |

### 工作负载管理 (WLM)

- 支持配置多个查询队列
- 每个队列可设置不同的优先级和并发数
- **静态队列**：固定资源分配
- **动态队列**：共享资源池

## S3 集成 - Redshift Spectrum

### Redshift Spectrum 功能

- 直接查询 **S3** 中的原生文件（Parquet、CSV、JSON 等格式）
- **无需将数据导入 Redshift 集群**
- 使用标准 SQL 查询 S3 数据
- 按扫描的数据量计费

### Spectrum 优势

| 优势 | 说明 |
|------|------|
| **成本效益** | 数据存储在低成本 S3，按需查询 |
| **无限扩展** | 不受 Redshift 集群存储限制 |
| **共享数据** | 多个集群可查询同一 S3 数据 |
| **格式支持** | 支持 Parquet（推荐）、ORC、Avro、CSV、JSON |

### Spectrum vs 传统加载

| 特性 | Spectrum | 传统加载 |
|------|---------|---------|
| 数据存储位置 | S3 | Redshift 集群 |
| 查询速度 | 较慢（需扫描 S3） | 快速（本地 SSD） |
| 成本 | 按扫描量计费 | 按集群规模计费 |
| 适用场景 | 低频查询、数据湖 | 高频查询、热数据 |

## 定价和成本优化

### 定价模型

| 计费项 | 说明 |
|--------|------|
| **节点费用** | 按运行时间和节点类型计费 |
| **快照存储** | 按存储量计费 |
| **Spectrum** | 按扫描的数据量计费 |
| **数据传输** | 跨区域数据传输收费 |

### 成本优化建议

1. **使用 Spectrum**：冷数据存储在 S3，按需查询
2. **压缩数据**：减少存储成本和 I/O
3. **选择合适的分布键**：减少数据移动，提升查询性能
4. **使用 Spot 实例**：开发测试环境使用
5. **暂停集群**：非生产环境使用 Auto-Stop
6. **选择合适的节点类型**：DS2 用于存储，DC2 用于计算

## 考试重点总结

### SAA-C02 高频考点

1. **列式存储**：Redshift 使用列式存储，适合分析查询
2. **分布策略**：KEY（大表 JOIN）、ALL（小表）、EVEN（默认）
3. **排序键**：减少 I/O，提升查询性能
4. **COPY 命令**：从 S3 批量加载数据到 Redshift
5. **Redshift Spectrum**：直接查询 S3 数据，无需导入
6. **与 RDS 区别**：RDS 是 OLTP，Redshift 是 OLAP
7. **快照备份**：自动快照存储在 S3
8. **压缩编码**：节省存储，提升性能
9. **MPP 架构**：并行处理，适合大规模数据分析
10. **WLM**：查询队列优先级管理

### 场景题解题思路

```
场景分析 → 选择解决方案
├── "需要分析多年销售数据" → Redshift
├── "需要实时查询用户信息" → RDS / DynamoDB
├── "需要查询 S3 中的历史数据" → Redshift Spectrum
├── "需要大规模数据仓库" → Redshift
└── "需要低延迟键值查询" → DynamoDB
```

## 最佳实践

1. **选择合适的分布键**：JOIN 频率高的列作为分布键
2. **使用排序键**：常用过滤列作为排序键
3. **启用数据压缩**：减少存储和 I/O
4. **使用 COPY 命令**：批量加载优于 INSERT
5. **Spectrum 用于冷数据**：热数据加载到 Redshift
6. **配置 WLM**：为重要查询分配高优先级
7. **监控查询性能**：使用 Query Monitor Rules
8. **定期快照**：配置自动快照和跨区域复制
