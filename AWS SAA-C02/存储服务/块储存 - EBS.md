# EBS - Elastic Block Store

> **EBS (Elastic Block Store)** 是一种为 EC2 实例提供的高性能、持久化块级存储设备。数据持久存储在独立于实例的存储后端，即使实例停止或终止，数据依然保留。
>
> 相关文档：[[存储服务 - Storage Service]] | [[文件储存 - AWS EFS]] | [[S3 - Object Storage]] | [[虚拟私有云 - VPC]]

---

## 核心概念

### 块存储 vs 对象存储

| 特性 | EBS (块存储) | [[S3 - Object Storage]] (对象存储) |
|------|-------------|--------------|
| 数据结构 | 将数据划分为块，由文件系统管理 | 扁平的命名空间，对象由键（Key）标识 |
| 访问方式 | 挂载为块设备（如 `/dev/sdf`） | 通过 REST API (HTTP/HTTPS) 访问 |
| 适用场景 | 操作系统盘、数据库、企业级应用 | 静态资源、备份归档、大数据分析 |
| 一致性模型 | 强一致性 | 最终一致性（列出操作） |

### 持久化存储

- EBS 卷是**独立于 EC2 实例的持久化存储**
- 创建时需**预先指定卷大小**（最小 1 GB）
- 实例终止后，卷数据默认**不会自动删除**（可通过 `DeleteOnTermination` 参数控制）
- 支持**跨挂载点迁移**：卸载卷后可挂载到其他实例

## 卷类型

### SSD 优化型（高性能）

| 卷类型 | 全称 | 适用场景 | 特性 |
|--------|------|---------|------|
| **io2 / io1** | Provisioned IOPS SSD | 关键业务数据库、高 IOPS 需求 | 最高 256,000 IOPS，可独立配置 IOPS 与容量 |
| **gp3** | General Purpose SSD | 通用场景（默认推荐） | 基准 3,000 IOPS，可独立调整 IOPS 和吞吐量 |
| **gp2** | General Purpose SSD | 中小规模工作负载 | 1-10,000 IOPS（与容量挂钩，1 GB = 1 IOPS，最高 10,000） |

### HDD 优化型（低成本）

| 卷类型 | 全称 | 适用场景 | 特性 |
|--------|------|---------|------|
| **st1** | Throughput Optimized HDD | 大数据、数据仓库、日志处理 | 低成本的顺序读写，最高 500 IOPS（1 MiB 操作） |
| **sc1** | Cold HDD | 低频访问、冷数据归档 | 最低成本，最高 250 IOPS |

> **gp3 的优势**：相比 gp2，gp3 提供更高的基准性能（3,000 IOPS / 125 MB/s 吞吐量），且 IOPS、吞吐量可独立于容量调整，平均节省 20% 成本。

## 高可用性与可靠性

- **单可用区内自动复制**：EBS 卷在单个可用区（AZ）内冗余存储多个副本
- **可用性指标**：提供 **99.8% ~ 99.99%** 的可用性（取决于卷类型）
- **数据丢失率**：每年每 PB 数据丢失率低于 **0.1%**（4 个 9 的可靠性）
- **跨可用区保护**：EBS 本身**不支持跨 AZ 复制**，需通过快照实现跨区域/可用区备份

---

## 数据保护与自动化

### EBS 快照

- 快照是 EBS 卷在**特定时间点**的增量备份
- 仅复制**已使用的数据块**，节省存储空间和成本
- 快照默认存储在 **[[S3 - Object Storage]]** 中（对用户透明）
- 支持跨区域快照复制（Cross-Region Copy）
- 快照也可用于创建 [[文件储存 - AWS EFS]] 的备份

### 数据生命周期管理器 (DLM)

- 自动化管理 EBS 快照的**创建、保留和删除**
- 可配置策略：
  - 快照频率（每小时/每天/每周）
  - 保留数量或保留时长
  - 自动删除过期快照
- 减少手动管理开销，符合**最少管理**最佳实践

## 安全与加密

### 静态数据加密

- EBS 与 **AWS KMS** 深度集成（与 [[S3 - Object Storage]] 和 [[文件储存 - AWS EFS]] 的加密方式一致）
- 支持使用 **AWS 托管密钥** 或 **客户托管密钥 (CMK)**
- 加密在存储层面透明进行，对应用程序无感知

### 传输中加密

- 支持 **EBS 加密** 在卷与实例之间的数据传输加密
- 需要实例类型支持加密功能

### 安全最佳实践

- 使用 IAM 策略控制快照和卷的访问权限
- 启用 CloudTrail 审计 EBS 和快照的 API 调用
- 使用 CMK 实现密钥轮换和细粒度权限控制

## 监控与指标

### CloudWatch 集成

- EBS 自动向 **CloudWatch** 发送性能指标（需先了解 [[CloudWatch]] 的基础概念）
- 核心指标：
  - `VolumeReadOps` / `VolumeWriteOps`
  - `VolumeReadBytes` / `VolumeWriteBytes`
  - `VolumeThroughputPercentage`
  - `VolumeQueueLength`
  - `BurstBalance`（gp2 卷特有）

### 指标状态说明

| 状态 | 说明 |
|------|------|
| `INSUFFICIENT_DATA` | 卷刚创建，CloudWatch 尚未收集足够数据 |
| `OK` | 指标正常，在阈值范围内 |
| `ALARM` | 指标超出设定的告警阈值 |

> **注意**：新创建的 EBS 卷需要等待几分钟才会开始产生 CloudWatch 数据。

## 性能限制

### EBS IOPS 存储限制（区域性）

EBS 限制是**区域级别**的，包括：

| 限制项 | 默认值 | 说明 |
|--------|--------|------|
| 总预配置 IOPS | 300,000 PIOPS | 区域内所有 PIOPS 卷的 IOPS 总和 |
| PIOPS 卷总容量 | 300 TiB | 区域内 PIOPS 卷的存储容量上限 |
| 常规 SSD 总容量 | 50,000 GiB | 区域内 gp2/gp3 卷的总容量 |

> 如需提高限制，可通过 **AWS Support Center** 提交限额提升请求。

## 管理要点

### 卷扩容

- EBS 支持**在线扩容**（无需停止实例）
- 支持 gp2、gp3、io1、io2、st1、sc1 类型
- 扩容后需在操作系统内部扩展文件系统（如 Linux 的 `growpart` + `resizefs`）

### 性能调整

- **gp3 / io2 / io1**：支持在线调整 IOPS 和吞吐量，无需停机
- **gp2**：扩容时 IOPS 随容量自动增加，但手动降低 IOPS 不可行

### 最佳实践

1. **优先使用 gp3**：相比 gp2 成本更低，性能更可独立调配
2. **使用 DLM 自动化快照**：减少手动管理成本
3. **启用加密**：默认启用 EBS 加密，确保所有新卷自动加密
4. **监控性能指标**：通过 CloudWatch 及时发现性能瓶颈
5. **跨 AZ/区域备份**：利用快照复制实现灾难恢复

### 与 AWS Snowball 的数据迁移

- 对于大规模数据迁移，可以使用 [[AWS Snowball]] 将本地块存储数据物理传输到 AWS
- 数据到达后，可先转换为 EBS 快照或存入 [[S3 - Object Storage]]
