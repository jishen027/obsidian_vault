# EFS - Elastic File System

> **EFS (Elastic File System)** 是一种基于 **NFS (Network File System)** 协议的无服务器、弹性可扩展的分布式文件存储服务。专为需要共享文件访问的跨实例协作场景设计。
>
> 相关文档：[[存储服务 - Storage Service]] | [[块储存 - EBS]] | [[S3 - Object Storage]] | [[虚拟私有云 - VPC]]

---

## 核心概念

### EFS vs EBS 对比

| 特性 | EFS (文件存储) | [[块储存 - EBS]] (块存储) |
|------|---------------|-------------|
| 访问方式 | NFS 协议，通过挂载点访问 | 块设备挂载（如 `/dev/sdf`） |
| 并发访问 | 支持数千个实例同时读写 | 单卷仅能挂载到单个 EC2 实例 |
| 文件系统 | 原生 POSIX 兼容 (Linux) | 需自行格式化 (ext4, xfs 等) |
| 扩展性 | 自动弹性伸缩，无需预分配容量 | 创建时需指定大小，扩容需手动操作 |
| 服务器架构 | 无服务器，无需管理 | 需手动管理和维护卷 |

### 多实例共享访问

- EFS 支持**数千个 EC2 实例同时挂载**并读写同一个文件系统
- 通过 **NFSv4** 协议实现并发访问
- 与 [[块储存 - EBS]] 不同，EBS 只能挂载到单个 EC2 实例
- 适用于：
  - 内容管理系统 (CMS)
  - 网站托管
  - 开发测试环境
  - 大数据分析工作负载
  - 容器化应用共享存储

## 高可用性与持久性

### 原生高可用

- EFS 默认在**多个可用区 (AZ)** 内自动存储数据副本
- 单可用区内数据冗余存储，提供**99.999999999% (11个9)** 的数据持久性
- 自动故障转移：单个可用区故障时，EFS 自动切换到其他可用区，无需手动干预

### 持久性 vs 临时存储

| 存储类型 | 持久性 | 数据保留 | 适用场景 |
|---------|--------|---------|---------|
| **EFS** | 持久化 | 实例终止后数据保留 | 共享文件、数据库备份 |
| **[[块储存 - EBS]]** | 持久化 | 实例终止后数据默认保留 | 操作系统盘、数据库 |
| **Instance Store** | 临时性 | 实例停止/终止后数据丢失 | 缓存、临时数据处理 |

## 存储类别与性能

### 存储类别

| 存储类别 | 访问延迟 | 成本 | 适用场景 |
|---------|---------|------|---------|
| **Standard (STANDARD)** | 毫秒级 | 较高 | 频繁访问的热数据 |
| **Infrequent Access (INFREQUENCY)** | 毫秒级 | 较低（比 Standard 低 92%） | 不频繁访问的冷数据 |

> **自动数据分层**：EFS 自动将文件从 Standard 移动到 Infrequent Access，无需应用程序修改。基于最近访问时间自动触发，通常 7 天未访问的文件会被移动到 IA 层。
>
> 与 [[S3 - Object Storage]] 的 Glacier 存储类类似，EFS IA 层也为冷数据提供显著的成本节省。

### 性能模式

| 性能模式 | 吞吐量 | 最大 IOPS | 适用场景 |
|---------|--------|----------|---------|
| **General Purpose** | 适中 | 最高 100,000 | 大多数应用场景（默认推荐） |
| **Max I/O** | 更高 | 基本无上限 | 大规模并行计算、高性能计算 |
| **Provisioned Throughput** | 可预测 | 与吞吐量关联 | 需要固定吞吐量的工作负载 |

> **General Purpose** 模式适用于大多数场景。如果性能不足，可通过 **EFS 自动缩放** 或切换到 **Max I/O** 模式提升性能。

### 吞吐模式

| 吞吐模式 | 说明 |
|---------|------|
| **Bursting** | 与存储容量关联，存储越多基准吞吐量越高 |
| **Provisioned** | 可独立预置吞吐量（1-4 GiB/s），适用于可预测的工作负载 |
| **Elastic** | 吞吐量随存储容量自动扩展 |

## 访问与安全

### 连接方式

- 实例通过 **[[虚拟私有云 - VPC]]** 内部连接 EFS，流量不经过公网
- 通过 **EFS 挂载点 (Mount Target)** 在子网中暴露 NFS 端点
- 支持 **跨 VPC 挂载**：使用 VPC Peering 或 Transit Gateway 实现
- 支持 **混合云架构**：通过 **AWS DataSync** 或 **EFS Access Points** 实现本地与云端的文件同步

### 网络安全

- 通过 **安全组** 控制对 EFS 挂载点的访问（参考 [[Security Group]] 和 [[NACL]]）
- 仅允许来自特定安全组或子网的 NFS 流量（端口 2049）
- 支持 **VPC Endpoints (Interface Endpoint)**：通过私有网络访问 EFS，不经过公网

### 数据加密

- EFS 与 **AWS KMS** 集成，支持静态数据加密（与 [[S3 - Object Storage]] 和 [[块储存 - EBS]] 的加密方式一致）
- 支持使用 **AWS 托管密钥** 或 **客户托管密钥 (CMK)**
- 传输中的数据可通过 **TLS** 加密

### 访问控制

- **IAM 策略**：控制对 EFS API 的访问（创建文件系统、挂载点等）
- **POSIX 权限**：文件系统内文件级别的 Linux 权限控制
- **EFS Access Points**：为应用程序定义统一的入口点，自动应用目录结构和 IAM 策略

## 数据保护与备份

### EFS 备份

- 使用 **AWS Backup** 实现自动化备份策略
- 支持：
  - 全量备份和增量备份
  - 跨区域备份复制
  - 备份保留策略（按天数或数量）
  - 从备份恢复到新文件系统
- 支持将数据备份到 [[S3 - Object Storage]] 或 [[S3 Glacier Archive]] 进行长期归档

### 数据保护最佳实践

1. **启用加密**：创建文件系统时启用 KMS 加密
2. **配置备份计划**：使用 AWS Backup 定期备份
3. **多 AZ 部署**：确保跨可用区冗余
4. **监控存储容量**：设置 CloudWatch 告警监控存储使用量

## 管理与成本

### 无服务器架构

- EFS 是**完全托管的无服务器服务**
- 无需预置容量或管理底层基础设施
- 存储容量自动扩展：从几 GB 到数 PB，无需干预
- 无需预置吞吐量：吞吐模式随使用量自动调整

### 成本优化

- **按实际使用量付费**：按存储容量和操作次数计费
- **利用自动数据分层**：将冷数据自动移动到 Infrequent Access 层降低成本
- **使用 Access Points**：避免不必要的文件扫描操作
- **监控存储增长**：设置预算告警防止意外费用

### 最佳实践

1. **跨多 AZ 部署**：利用多可用区实现高可用
2. **使用 Access Points**：简化应用访问，增强安全性
3. **启用自动数据分层**：降低冷数据存储成本
4. **配置 AWS Backup**：实现自动化备份和灾难恢复
5. **通过安全组限制访问**：仅允许必要的 NFS 流量（参考 [[Security Group]]）

### 与 AWS Snowball 的数据迁移

- 对于大规模数据迁移，可以使用 [[AWS Snowball]] 将本地数据物理传输到 AWS
- 数据到达后，可先存入 [[S3 - Object Storage]]，再迁移到 EFS
