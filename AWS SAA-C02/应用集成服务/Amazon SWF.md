# Amazon SWF - Simple Workflow Service

> **Amazon SWF (Simple Workflow Service)** 是一种完全托管的应用集成服务，用于协调分布式应用组件之间的任务执行。它专门设计用于管理长时间运行的复杂业务流程。
>
> 相关文档：[[SQS]] | [[Amazon API Gateway]]

---

## 核心概念

### SWF 定位

| 服务 | 类型 | 场景 | 特点 |
|------|------|------|------|
| **SWF** | 工作流编排 | 复杂业务流程协调 | 状态跟踪、人工干预 |
| **SQS** | 消息队列 | 应用解耦、异步处理 | 消息缓冲、削峰填谷 |
| **Step Functions** | Serverless 工作流 | 无服务器工作流编排 | 与 Lambda 深度集成 |
| **SNS** | 消息通知 | 发布/订阅消息分发 | 广播通知 |

### SWF vs Step Functions 对比（考试重点）

| 特性 | Amazon SWF | AWS Step Functions |
|------|-----------|-------------------|
| **架构模型** | 需要工作者（Worker） | 无服务器（Lambda 原生） |
| **状态管理** | 服务自动跟踪 | 服务自动跟踪 |
| **执行时间** | 最长 1 年 | 最长 1 年 |
| **人工干预** | 原生支持 | 支持（Human Task） |
| **适用场景** | 传统应用、需要工作者 | Serverless、Lambda 驱动 |
| **成本** | 较高（需要工作者） | 按执行次数计费 |

> **考试提示**：AWS 推荐使用 **Step Functions** 作为新的工作流编排服务，SWF 适用于需要工作者（Worker）的遗留系统。

## 核心架构

### SWF 核心组件

| 组件 | 描述 | 说明 |
|------|------|------|
| **Workflow Type** | 工作流类型定义 | 定义工作流的逻辑和步骤 |
| **Workflow Instance** | 工作流实例 | Workflow Type 的具体执行 |
| **Activity Type** | 活动类型定义 | 定义要执行的任务 |
| **Activity Task** | 活动任务 | 分发给工作者执行的具体任务 |
| **Decision Type** | 决策类型 | 决定下一步执行什么 |
| **Decision Task** | 决策任务 | 发送给决策者的状态检查 |

### SWF 工作流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  决策者      │←────│  SWF 服务    │←────│  工作者      │
│ (Decider)   │────→│             │────→│ (Worker)    │
└─────────────┘     └─────────────┘     └─────────────┘
       │                                       │
       └──── 记录状态和进度 ←───────────────────┘
```

### 执行流程

```
1. 启动工作流 → SWF 创建 Workflow Instance
2. SWF 发送 Decision Task → Decider 获取工作流状态
3. Decider 做出决策 → 注册 Activity Tasks
4. SWF 分配 Activity Task → Worker 执行任务
5. Worker 报告结果 → SWF 记录状态
6. 重复步骤 2-5 直到工作流完成
```

## 核心功能

### 任务协调

- 确保复杂业务逻辑中的各个步骤按正确顺序执行
- 支持并行任务执行
- 支持条件分支和循环

### 状态管理和容错

| 功能 | 说明 |
|------|------|
| **状态跟踪** | 持续跟踪每个任务的状态（成功、失败、进行中） |
| **进度记录** | 记录工作流当前进度 |
| **断点续传** | 组件崩溃后从断点继续 |
| **自动重试** | 失败任务自动重试 |

### 人工干预（特色功能）

- 支持将**人工事件**集成到自动化流程中
- 工作流可以等待人工操作（如管理员点击"已发货"）
- 适用于需要人类审核的业务流程

### 应用解耦

- 将业务流程控制逻辑从应用代码中抽离
- 独立修改业务流而不改动底层微服务代码
- 支持分布式应用组件协作

## 典型应用场景

### 场景示例

| 场景 | 流程说明 | SWF 优势 |
|------|---------|---------|
| **电子商务订单处理** | 下单 → 支付验证 → 库存检查 → 发货 → 通知 | 支持人工审核订单 |
| **视频资产转换** | 上传 → 转码 → 截图 → 存储 → 通知 | 长时间运行、多步骤 |
| **贷款审批** | 申请 → 信用检查 → 人工审批 → 放款 | 需要人工干预 |
| **媒体处理流水线** | 上传 → 元数据提取 → 转码 → 分发 | 并行处理、状态跟踪 |

### 适用条件

| 条件 | 说明 |
|------|------|
| **长时间运行** | 工作流可运行数小时到 1 年 |
| **多步骤协调** | 需要按特定顺序执行多个任务 |
| **需要人工干预** | 流程中包含人工审核步骤 |
| **需要状态跟踪** | 需要精确跟踪每个步骤的状态 |
| **容错要求高** | 组件故障时需要自动恢复 |

## 与 SQS 对比

### 消息队列 vs 工作流

| 特性 | SQS | SWF |
|------|-----|-----|
| **主要用途** | 应用解耦、消息缓冲 | 任务协调、工作流编排 |
| **状态管理** | 不跟踪消息状态 | 自动跟踪任务状态 |
| **消息处理** | 处理一次（At-Least-Once） | 确保处理完成 |
| **重试机制** | 死信队列 | 自动重试 |
| **适用场景** | 解耦、削峰 | 复杂业务流程 |

### 选择指南

```
需要协调多个任务按顺序执行？
├── 是 → 需要人工干预？
│         ├── 是 → SWF
│         └── 否 → Step Functions
└── 否 → 需要应用解耦？
          ├── 是 → SQS
          └── 否 → 直接调用
```

|--------|-----|

| **决策名称长度** | 256 字符 |

## 定价模型

### 计费项

| 计费项 | 说明 |
|--------|------|
| **活动任务** | 按创建的任务数量计费 |
| **决策任务** | 按创建的决策任务数量计费 |
| **数据存储** | 按存储的数据量计费 |
| **工作者** | 工作者运行成本（EC2、ECS 等） |

### 成本优化建议

1. **考虑 Step Functions**：Serverless 架构可能更经济
2. **优化工作者**：使用 Spot Instances 运行工作者
3. **合理设置超时**：避免任务长时间挂起
4. **批量处理**：合并小任务减少计费次数

## 考试重点总结

### SAA-C02 高频考点

1. **SWF 核心功能**：协调分布式应用组件之间的任务
2. **状态管理**：SWF 自动跟踪任务状态，提供容错能力
3. **人工干预**：SWF 支持等待人工操作（特色功能）
4. **与 SQS 区别**：SWF 协调任务，SQS 消息队列
5. **与 Step Functions 区别**：SWF 需要工作者，Step Functions 无服务器
6. **长时间运行**：支持最长 1 年的工作流执行
7. **断点续传**：组件故障后从断点继续
8. **应用解耦**：将业务逻辑从应用代码中抽离

### 场景题解题思路

```
场景分析 → 选择集成服务
├── "协调复杂业务流程" → SWF / Step Functions
├── "需要人工干预" → SWF
├── "Serverless 工作流" → Step Functions
├── "应用解耦、消息缓冲" → SQS
├── "发布/订阅通知" → SNS
└── "长时间运行、状态跟踪" → SWF
```

### 常见考试场景分析

**场景：公司需要构建一个订单处理系统，涉及支付验证、库存检查、人工审核和发货通知，需要精确跟踪每个步骤的状态，最具成本效益的方案？**

| 选项 | 是否正确 | 原因 |
|------|---------|------|
| **Amazon SWF** | ✅ 正确 | 协调复杂流程、支持人工干预、自动状态跟踪 |
| SQS | ❌ 错误 | 消息队列，不协调任务顺序，不跟踪状态 |
| SNS | ❌ 错误 | 发布/订阅服务，不适合复杂工作流 |
| EC2 | ❌ 错误 | 虚拟机，需要自行管理工作流状态 |

**关键总结：**
- **"协调复杂流程"** → SWF / Step Functions
- **"需要人工干预"** → SWF
- **"应用解耦"** → SQS
- **"通知分发"** → SNS
- **"Serverless 工作流"** → Step Functions

---

## 最佳实践

1. **选择合适的服务**：复杂流程用 SWF/Step Functions，消息队列用 SQS
2. **设置合理超时**：避免任务长时间挂起
3. **实现健康检查**：监控工作者状态
4. **使用死信队列**：处理无法成功执行的任务
5. **考虑 Step Functions**：新项目优先考虑无服务器方案
6. **记录详细日志**：便于问题排查和审计
7. **人工任务设置提醒**：确保人工步骤及时处理
